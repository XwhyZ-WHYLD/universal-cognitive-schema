# RFC-0001 — Universal Cognitive Schema: Core Specification

**Status:** Draft
**Created:** 2026-04-07
**Author:** Roshan George Thomas (rgthomas007)
**Repository:** https://github.com/XwhyZ-WHYLD/universal-cognitive-schema

---

## Abstract

This RFC defines the Universal Cognitive Schema (UCS) — an open, versioned, platform-agnostic data standard for representing a person's accumulated AI interaction profile. UCS enables cognitive identity portability across AI platforms without requiring cooperation from any single vendor.

---

## Problem Statement

Every major AI platform — OpenAI, Anthropic, Google, Meta — maintains a user's cognitive history in a proprietary format. When a user switches platforms, they lose all accumulated context: their communication preferences, domain expertise signals, interaction patterns, and project history. There is no standard for what "AI cognitive identity" means, and no platform has an incentive to create one.

This creates compounding lock-in. The longer a user invests in a platform, the harder it becomes to leave. This is not a feature gap — it is a structural problem that platforms will never solve on their own.

Only a neutral, open, user-owned standard can solve this. UCS is that standard.

---

## Goals

1. Define a portable, machine-readable format for AI cognitive identity
2. Enable lossless or near-lossless migration between compliant AI platforms
3. Give users ownership and control of their cognitive data
4. Remain platform-neutral — no single company controls the standard
5. Support extensibility without breaking core interoperability

---

## Non-Goals

- UCS is not a product or service
- UCS is not a centralised database
- UCS does not require platform cooperation to function (data export APIs already exist via GDPR mandates)
- UCS does not prescribe how platforms store data internally

---

## Schema Overview

A UCS profile is a structured JSON document with seven core dimensions:

| Dimension | What it captures |
|---|---|
| `persona` | Communication style, formality, verbosity, tone markers |
| `expertise_map` | Domains of knowledge and depth calibration |
| `project_graph` | Active and archived projects with context |
| `preference_corpus` | Output format preferences, response length defaults |
| `interaction_patterns` | Prompt style, corrections, reinforced/rejected behaviours |
| `trust_boundaries` | Delegation tolerance, sensitive domains, confirmation preferences |
| `temporal_context` | Recency weights, life chapters, evolution over time |

Each dimension is optional except `persona` and `provenance`. A minimal valid profile requires only a persona and provenance record.

---

## Key Design Decisions

### D1 — Weighted communication style (not enum)

`persona.communication_style` is a weighted dict (e.g. `{"direct": 0.7, "collaborative": 0.3}`) rather than a single enum value. This reflects that real humans blend styles. A single label loses information that matters for accurate Echo construction.

*Trade-off:* slightly more complex to implement; significantly more accurate in practice.

### D2 — Numeric formality and verbosity (0.0–1.0)

Rather than string enums ("casual", "formal"), formality and verbosity are floats. This allows fidelity scoring to reflect partial signal confidence and enables smooth interpolation as more interaction data accumulates.

*Trade-off:* less human-readable at a glance; more machine-useful for constraint engine tuning.

### D3 — Provenance as a required field

Every profile must carry a `provenance` record: source platforms, extraction method, sanitisation attestation, and optional cryptographic signature. This makes the trust chain explicit and auditable — essential for the hardened architecture.

### D4 — Sanitisation attestation mandatory

Per the UCS security model, every profile passing through a compliant tool must pass through a sanitisation layer that strips prompt-injection patterns, role-override syntax, and executable instructions. The `sanitised` boolean and `sanitised_at` timestamp in provenance record this attestation.

### D5 — Extensions namespace

The `extensions` field allows platform-specific or community extensions without breaking core interoperability. Extension keys must be namespaced (e.g. `com.echonet.display_name`). Readers ignore unknown extensions rather than erroring.

## Fidelity Reporting

UCS is honest about what it can and cannot capture. Every profile export must include a fidelity report indicating confidence scores (0.0–1.0) per dimension. This prevents false precision and tells users exactly what migrated well and what didn't.

Example:
```json
{
  "fidelity": {
    "persona": { "score": 0.91, "note": "High confidence — inferred from 2,400 interactions" },
    "trust_boundaries": { "score": 0.55, "note": "Low — insufficient signal in source data" }
  }
}
```

## Security Requirements

1. All profiles must be encrypted at rest and in transit in compliant implementations
2. Sanitisation attestation is a required field — no compliant tool may skip this step
3. No compliant implementation may transmit a decrypted profile to a third party without explicit, per-dimension user consent
4. The `attestation_signature` field supports optional cryptographic signing of the provenance record

See `SECURITY.md` and `hardened_spec.md` for the full threat model and 10-layer defensive architecture.

## Open Questions

The following questions are open for community input:

**OQ-1: Schema versioning strategy**
How should breaking changes to the schema be handled? Options: semver with strict compatibility guarantees; graceful degradation (v2 profiles degrade in v1 readers); or a migration layer. Current preference: graceful degradation with explicit `ucs_version` field.

**OQ-2: Interoperability testing**
What constitutes a compliant UCS implementation? Should there be a conformance test suite? Who maintains it? Current preference: community-maintained test suite in `tests/`, conformance badge for implementations that pass.

**OQ-3: Governance structure**
The standard is currently maintained by a single contributor. As adoption grows, governance should move to a foundation or working group. Timeline and structure TBD. Input welcome.

**OQ-4: Biometric anchoring for Echoes**
The EchoNet reference implementation (RFC-0002) proposes cryptographic linking of Echo profiles to biometric signatures for authenticity verification. Should the core UCS spec define a slot for this, or leave it to the EchoNet extension namespace?

**OQ-5: Temporal decay functions**
`temporal_context.recency_weight` defines how much recent interactions dominate. Should UCS specify standard decay functions (exponential, linear, step) or leave implementation to the parser?

## Reference Implementation

The `ucs_parser` module provides a reference implementation:

```bash
pip install ucs-parser
```

```python
from ucs_parser import Parser, Validator

# Parse a ChatGPT export
parser = Parser(source="chatgpt")
profile = parser.from_file("chatgpt_export.zip")

# Validate
validator = Validator()
result = validator.validate(profile.to_dict())

# Fidelity report
print(profile.fidelity_report())
```

Stage 1 MVP tools (`capture_profile.py`, `stage1_mvp.py`) demonstrate manual profile creation and Echo query execution.
