**Status:** Draft

**Author:** Roshan George Thomas

---

## 1. Summary

RFC-0004 defines **Governance v1** for UCS / EchoNet profiles. It specifies the
rules and data structures that determine *who controls a cognitive profile*,
*how it may be deleted*, *what happens to it after the principal's death*, and
*how disputes over it are resolved*.

Governance v1 is a **witness and gate, not an owner**. It never silently
rewrites cognitive content. It records ownership, evaluates whether a requested
action is permitted, emits a tamper-evident audit event, and exposes a single
decision function the runtime can enforce.

Governance v1 deliberately defines the *policy and data shape* for posthumous
use and inheritance; it does **not** implement the Stage 6 *execution* engine
(operating a successor's Echo). It writes the rules under which such execution
would later be permitted.

---

## 2. Motivation

UCS already supports cryptographic identity (RFC-0003) and runtime delegation
consent (the `trust_boundaries` field). Neither answers the questions that
become urgent the moment an Echo is monetizable and deployable by third
parties:

- Who owns a profile, and who may act on the owner's behalf?
- How is a profile deleted, and is deletion reversible or final?
- What happens to an Echo when its principal dies â€” does it stop, become a
  read-only memorial, or keep operating commercially?
- When two parties dispute a profile (impersonation, ownership, accuracy), how
  is the conflict adjudicated without an authority silently seizing the data?

These gaps are not cosmetic. An EchoNet Echo can earn revenue and make
representations. The one party who can no longer object â€” the deceased â€” must
have opted in **explicitly** before the commercial posthumous path opens. This
asymmetry is the central design driver of Governance v1.

  "lifecycle_state": "active",
  "ownership": { "owner_did": "did:ucs:â€¦", "controllers": [], "transfer_locked": true },

Governance v1 is orthogonal to, and references, existing layers:

|---|---|

| Trust score and tier (T0â€“T5) | `trust_score.py` |

## 4. Lifecycle States

A profile is always in exactly one lifecycle state:

| State | Meaning |
|---|---|
| `active` | Operated by the living principal. All actions permitted. |
| `suspended` | Temporarily frozen (pending deletion, escalated dispute, security). Admin-only. |
| `deceased_pending` | Death asserted but not confirmed. Admin-only. |
| `posthumous` | Death confirmed; the posthumous mode governs what is permitted. |
| `deleted` | Tombstoned; cognitive content erased. |

State transitions are deliberate and logged. In particular, a single death
*assertion* never flips a profile straight into posthumous operation â€”
**confirmation is a separate step** so one false assertion cannot activate a
living person's Echo.

---

## 5. Ownership

- The **principal owns by default** (`owner_did` = the profile DID).
- The owner MAY grant **controllers** scoped authority (e.g. an executor with
  `lifecycle` scope, a custodian with `delete` scope). Controllers act on the
  owner's behalf **within scopes** and can never alter ownership.
- **Ownership transfer is locked by default** (`transfer_locked: true`) and
  requires the current owner plus an explicit unlock. Transfer is forbidden
  while a profile is `suspended` or posthumous â€” those paths go through
  inheritance, not transfer.

Scopes used by v1: `lifecycle`, `delete`, `admin`, `infer`. A controller with
`*` has all scopes.

---

## 6. Posthumous Use â€” the loaded path

Governance v1 defines **three posthumous modes**:

| Mode | Capability | Requirements |
|---|---|---|
| `dormant` (default) | Frozen. Administer-only (export, delete). | None â€” this is the default on any death event. |
| `memorial` | Read-only inference, **non-transacting**, mandatory disclosure label, successor-revocable. No new factual assertions attributed to the principal. | `inheritance.inheritable = true` **and** a designated `successor_id`. |
| `active` | May transact / make new representations. | All of `memorial`'s requirements **plus** a signed pre-mortem **directive** from the principal **plus** trust tier **â‰¥ T2** at time of death. |

**Resolution rules** (implemented in `governance.confirm_posthumous`):

1. Any registered **objector** forces `dormant`, overriding everything. An
   objection is a brake, not a claim of ownership; it grants the objector no
   operating rights.
2. A request for `memorial` without `inheritable` + successor is **downgraded
   to `dormant`**.
3. A request for `active` without a directive, or below the minimum tier, is
   **downgraded to `memorial`** (never silently denied â€” the principal's
   inheritability intent is honoured at the safe level).
4. An objector added *after* posthumous confirmation immediately re-freezes the
   Echo to `dormant`.

**Disclosure.** Any posthumously-operating Echo (memorial or active) MUST surface
a disclosure label, e.g.:

> *This Echo operates posthumously and does not represent the live intentions of
> {name}. Mode: {memorial|active}.*

The runtime obtains this from `governance.disclosure_label(profile)`.

**Why this asymmetry.** Memorial use (a family keepsake that answers but does not
transact) is humane and low-risk. Active use can earn money and make
representations under a successor's commercial incentive â€” so it requires the
deceased's own signed, pre-mortem consent and a verified-identity tier. The
default everywhere is the most conservative option.

## 7. Deletion

| Mode | Effect | Gate |
|---|---|---|
| `soft` | Reversible within a grace window (default 30 days). Profile â†’ `suspended`. Content retained. | `delete` scope. |
| `hard` | Cognitive content erased. Tombstone **and audit log retained**. Profile â†’ `deleted`. | `delete` scope. |
| `purge` | Everything erased **including the audit log**. Minimal tombstone only. | Owner, **or** `reason = "legal_order"` (e.g. a regulatory erasure order). |

Erasable cognitive fields include `persona`, `expertise_map`, `project_graph`,
`preference_corpus`, `interaction_patterns`, `temporal_context`, and the
profile-schema equivalents. Identity roots and the tombstone survive hard
deletion so that a deleted DID cannot be silently re-registered as if fresh.

## 8. Disputes

A dispute has a category (`impersonation`, `ownership`, `accuracy`,
`posthumous_objection`, `data_rights`, `other`), a claimant DID, a statement,
optional evidence references, and a state machine:

```
open â†’ under_review â†’ {resolved_upheld | resolved_rejected}
                    â†˜ withdrawn
```

- **Filing** a dispute does **not** freeze a profile â€” frivolous filings cannot
  weaponise suspension.
- **Escalation** moves it to `under_review` and MAY `suspend` the profile.
- **Resolution** restores the profile to `active` **only if no other escalated
  dispute is still holding the freeze**.

---

## 9. Tamper-Evident Audit Log

Every state transition appends an event to `governance.event_log`. Each event
embeds the SHA3-512 hash of the previous event, forming a hash chain:

```
event_n.prev_event_hash == sha3_512(canonical_json(event_{n-1}))
```

`governance.verify_event_chain(profile)` returns `false` if any historical event
was altered. This lets disputes be reconstructed and retroactive tampering be
detected. (A `purge` deliberately empties the log; the resulting tombstone
records that a purge occurred.)

---

## 10. The Central Gate

The entire system asks one question through one function:

```python
can_operate(profile, action="infer" | "transact" | "admin") -> {
    "allowed": bool, "reason": str, "disclosure": str | None, "state": str
}
```

This is the single enforcement point for EchoNet runtime, Jarvis (Stage 5b), and
any external integrator (Stage 8). It collapses all of the above â€” lifecycle
state, posthumous mode, suspension, deletion â€” into an enforceable decision plus
any required disclosure label.

## 12. Security & Ethical Considerations

- **Fail closed.** When the trust tier cannot be verified (trust engine
  unavailable), `active` posthumous operation is **denied**, not assumed.
- **No silent content mutation.** Governance never edits cognitive fields except
  via an explicit, logged deletion.
- **Objection always wins.** A named objector can force `dormant` at any time.
- **Jurisdictional note.** Post-mortem publicity / personality rights vary widely
  by jurisdiction, and impersonation/deception rules apply to active posthumous
  operation. Governance v1's disclosure label and directive requirement are
  necessary engineering controls but **not** a substitute for jurisdiction-
  specific legal review before any production posthumous deployment.

Acceptable language: *witness, gate, revocable, tamper-evident, fail-closed.*
Not acceptable: *legally binding, permanent ownership, guaranteed consent.*

## 13. Compatibility & Migration

- Profiles without a `governance` block are treated as `active` / self-owned
  until `init_governance` is called; `capture_profile.py` SHOULD attach a default
  block at creation (Stage 3.1 follow-up).
- The block is additive â€” it does not modify `provenance`, `trust_boundaries`, or
  `inheritance`. The existing `inheritance.release_condition = "post_mortem"`
  enum is the trigger Governance v1 reads when resolving posthumous mode.

## 14. Open Questions (deferred to Stage 3.1 / 6)

1. **Directive signature verification.** v1 stores a directive hash and trusts
   upstream signing; a verifier mirroring `attest.py` should validate the
   principal's signature over the directive.
2. **Distributed revocation registry.** `revocation_pointer` is defined
   (RFC-0003) but no registry yet resolves it; dispute and key revocation should
   share it.
3. **Multi-party / institutional ownership.** v1 models a single owner plus
   scoped controllers; organisational co-ownership is a Stage 7 concern.
4. **Inheritance execution.** Actually operating a successor Echo (vs. permitting
   it) is Stage 6 (Inheritance Lite).
