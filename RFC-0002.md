## 1. Abstract

This RFC defines EchoNet as the first reference consumer implementation of the Universal Cognitive Schema (UCS) protocol. EchoNet demonstrates that UCS profiles can power a real-world platform where human cognitive identity is deployed, scaled, and inherited — without the human being physically present.

EchoNet is not part of the UCS core protocol. It is a reference application that validates the protocol's design by using every layer of the UCS stack in production.

---

## 2. Motivation

UCS defines a standard for portable AI cognitive identity. But a protocol without a reference implementation is just a document. EchoNet serves three purposes:

1. **Validation** — proves UCS profiles are sufficient to power a real product
2. **Demonstration** — shows developers what a UCS-native application looks like
3. **Adoption** — creates a consumer surface that drives UCS profile creation at scale

---

## 3. What Is EchoNet

EchoNet is a platform where anyone can:

- **Create** their Echo — an AI surrogate trained on their UCS cognitive profile
- **Deploy** their Echo to work, advise, negotiate, and respond on their behalf
- **Discover** other Echoes in the network and hire them for specific tasks
- **Inherit** cognitive capital from Echoes of deceased persons

The Echo is not a generic chatbot. It is a UCS-profile-driven AI that behaves consistently with its human owner's documented cognitive identity — verified through the Trust Fabric.

## 5. The Jarvis Integration Layer

EchoNet's primary adoption mechanism is the Jarvis Integration Layer (UCS Stage 5b). Rather than asking users to consciously create their Echo, EchoNet deploys a Jarvis — a personal AI task assistant — that builds the Echo invisibly from natural usage.

**The core insight:**

> Don't ask someone to become an Echo. Let them use a Jarvis. The Echo crystallises from behaviour.

Every Jarvis interaction is captured by the UCS `interaction_patterns` field. The Echo improves automatically without the user doing anything. This solves the cold-start adoption problem that every identity platform faces.

The flywheel:

```
Jarvis adoption
      ↓
Echo created invisibly from usage
      ↓
UCS profile validated by real behavioural data
      ↓
EchoNet network grows organically
      ↓
More Jarvis users attracted by Echo value
      ↓
(repeat)
```

## 12. Open Questions

**OQ-1: Echo personhood**
At what trust tier, if any, should an Echo be granted legal standing to enter contracts on behalf of its human? This is a Stage 5 governance question, deferred to RFC-0005.

**OQ-2: Deceased Echo governance**
Who governs an Echo after its human dies? What can the Echo say, do, or refuse? Deferred to Stage 6 inheritance governance.

**OQ-3: Echo network effects**
Should Echoes be able to attest to each other's authenticity? Does an Echo attestation carry the same weight as a human attestation? Deferred to Stage 4 multi-Echo expansion.

**OQ-4: Jarvis open source strategy**
Should the Jarvis module be fully open source, or should EchoNet maintain a proprietary advantage at the Jarvis layer while keeping UCS open? Input welcome.

---

## 13. Relationship to UCS Core Protocol

EchoNet depends on UCS but UCS does not depend on EchoNet. Any developer can build an alternative consumer implementation on top of UCS. EchoNet's role is to:

1. Demonstrate that UCS is sufficient to power a real product
2. Provide a reference codebase that other implementors can study
3. Generate real-world usage data that improves the UCS specification

UCS is the passport. EchoNet is one country that accepts it. Many others can exist.
