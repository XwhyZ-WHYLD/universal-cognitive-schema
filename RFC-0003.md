# RFC-0003: UCS Trust Fabric v3

**Status:** Draft
**Stage:** 2
**Created:** 2026-05-24
**Protocol Area:** Identity, Provenance, Trust, Attestation

---

## 1. Summary

RFC-0003 introduces UCS Trust Fabric v3 вҖ” a quantum-ready, synthetic-resilient identity continuity layer for UCS Echo profiles.

The purpose of Trust Fabric v3 is not to prove perfect identity. Its purpose is to establish persistent, probabilistic, revocable, and cryptographically verifiable continuity between an Echo profile and its claimed human or institutional origin.

---

## 2. Motivation

UCS profiles currently support provenance metadata, but the existing provenance model does not fully support:

- DID creation and keypair generation
- Peer attestation with cryptographic signatures
- Revocation, rotation, and epoch tracking
- Temporal drift monitoring (TSSI)
- Synthetic identity risk scoring
- Post-quantum upgrade paths

Trust Fabric v3 fills this gap.

---

## 3. Identity Anchor

Every UCS profile SHOULD have a DID generated from an Ed25519 public key:

```
did:ucs:<sha3-256(public_key)>
```

The initial implementation uses Ed25519. Future implementations SHOULD support hybrid classical and post-quantum signatures:

- Ed25519 вҖ” classical signatures
- ML-DSA вҖ” post-quantum signatures
- SLH-DSA вҖ” recovery/root signatures
- ML-KEM вҖ” key establishment
- SHA3-512 / SHAKE256 вҖ” hashing

---

## 4. Provenance v3

The existing `attestation_signature` field is replaced by an `attestations` array. Each attestation SHOULD include:

- `type` вҖ” peer, institutional, legacy
- Issuer or attestor DID
- Signed profile hash (SHA3-512 of canonical JSON)
- Signature algorithm
- `created_at` and `expires_at` timestamps
- `revocation_pointer`

Legacy `attestation_signature` values are preserved as `type: legacy` entries.

## 7. Temporal Stewardship (TSSI)

Trust Fabric v3 includes a TSSI-inspired witness layer. This layer records drift across five dimensions вҖ” behavior, claims, values, security, attestation вҖ” without governing, banning, or deleting profiles.

The TSSI layer is a witness, not an authority.

## 9. Revocation

All keys and attestations SHOULD support:
- Expiry via `expires_at`
- Revocation via `revocation_pointer`
- Key rotation via `key_epoch`
- Epoch tracking

---

## 10. Security Considerations

Threats addressed: replay attacks, key compromise, credential forgery, Sybil manipulation, synthetic identity attacks, AI-text mimicry, deepfake impersonation, long-term behavioral drift.

Acceptable language: *quantum-ready, tamper-evident, synthetic-resilient, revocable.*
Unacceptable language: *immune to all attacks.*

## 12. Quick Start

```bash
pip install cryptography

# Generate trust anchor
python trust_anchor.py --out alice_anchor.json

# Migrate existing profile to v3 provenance
python provenance_v3.py \
  --profile echo_profile.json \
  --anchor alice_anchor.json \
  --out echo_profile_v3.json

# Create peer attestation
python attest.py \
  --profile echo_profile_v3.json \
  --anchor alice_anchor.json \
  --attestor-name "Alice Example" \
  --statement "I know this Echo owner and confirm this Echo behaves consistently with them." \
  --out peer_attestation.json
```

---

## 13. Compatibility

Existing UCS profiles using provenance v0.1.0 MAY be migrated to v3 using `provenance_v3.py`. Legacy `attestation_signature` values are preserved as legacy attestation entries.
